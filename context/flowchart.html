☰ Phase Breakdown

1
Intake

An idea or client request arrives from the CX team, Sales team, or as an internal product idea. All sources are funneled into a single validation pipeline.

2
Strategic Fit Check

Evaluate how the feature can be implemented within our system, then run it through a strategic alignment check: Does it align with company OKRs? Does it match the company strategy? Does it support revenue or growth goals? If any answer is no, the idea is added to the backlog for later consideration rather than discarded outright.

↓
Backlog If Not Aligned

Ideas that don't currently align with OKRs, company strategy, or revenue/growth goals are parked in the backlog. They can be revisited in future planning cycles when priorities shift.

3
Product Fit

Determine product fit — ensure the feature is usable and valuable across the entire client base, not just a niche segment. If not broadly applicable, explore how to adapt it for wider use.

4
Scope Check

Determine whether the validated idea is large enough to warrant full market research and competitor analysis. Smaller features skip directly to story writing and UX review, keeping the process lean.

5
Market Research If Large

Analyze how competitors price similar features and how that pricing is calculated. Identify which competitor plans include the feature. Map competitor pricing and plan structures to our own system to ensure alignment.

6
Competitor Analysis If Large

Compare feature scope, depth, and positioning against competitors. Identify gaps we can fill and differentiators we can leverage. Decide whether the idea is validated or should be discarded/parked.

7
Story Writing

Use the Lovable project to draft the user story. Define clear acceptance criteria, include pricing and plan mapping details, and document expected behavior for all client segments.

8
UX Review If Needed

Meet with a UX/Design expert to review user flows and wireframes. Define the simplest, best user experience while ensuring consistency with existing UI patterns.

9
Tech Review If Large Task

Discuss with the tech team to review system architecture impact, validate technical requirements and constraints, identify dependencies and risks, and get an effort/time estimation.

10
Development

The tech team builds the feature — code implementation, peer code review, and internal QA/testing before marking development as complete.

11
UAT

Product team performs User Acceptance Testing: verify against acceptance criteria, confirm pricing and plan integration works correctly, and test across different client segments. If issues are found, they loop back to development.

12
Deployment

Once UAT passes, give the green light to the tech team to deploy the feature to production.

13
Help Center

Write a help center article covering how the feature works and how clients can use it, including usage instructions and screenshots. Publish the article to the help center for client self-service.

14
Release Notes

By end of week, publish a release note covering all released features with descriptions and usage guidance.
